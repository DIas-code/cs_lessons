Part 2 — OOP

L10 — OOP pt.1: Classes, Objects, Constructors

10 April 2026 · Friday · 80 min Today

Object-Oriented Programming is the foundation of modern C# development. This lesson introduces the core building blocks: what objects really are, how to design classes that model real-world entities, and how constructors initialise them. You will shift from thinking about code as isolated functions to thinking about collections of objects that collaborate.

🎯 Learning Objectives

  • Define what OOP is and explain why it replaced purely procedural code
  • Distinguish between a class (blueprint) and an object (instance) with concrete examples
  • Create fields, auto-properties, validated properties, and computed properties
  • Build constructors — default, parameterised, chaining, and primary (C# 12)
  • Apply access modifiers to enforce encapsulation and data integrity
  • Explain value-type vs reference-type behaviour for class instances

📖 Theory

1. What Is OOP and Why It Exists

In Part 1 you wrote procedural code — data lives in separate variables and functions operate on them from the outside. This works at small scale, but with 30 students you'd need 30 names, 30 ages, 30 GPAs — all disconnected.

The procedural problem — scattered data
C# — Procedural: data and behaviour are separate Program.cs 
string studentName = "Aisha";
int    studentAge  = 20;
double gpa         = 3.8;

void PrintStudent(string n, int a, double g)
{
    Console.WriteLine($"{n}, age {a}, GPA {g}");
}

PrintStudent(studentName, studentAge, gpa);

OOP solves this by bundling related data and the functions that work on it into a single unit called an object. A student becomes one entity that knows its own name, age, and GPA — and can print itself.

The OOP solution — data and behaviour together
C# — OOP: everything in one place Program.cs 
class Student
{
    public string Name { get; set; }
    public int    Age  { get; set; }
    public double GPA  { get; set; }

    public void Print()
        => Console.WriteLine($"{Name}, age {Age}, GPA {GPA}");
}

Student s = new Student { Name = "Aisha", Age = 20, GPA = 3.8 };
s.Print();  // Aisha, age 20, GPA 3.8

C# is object-oriented from the ground up. Every type you already used — string, Console, List<T> — is a class. OOP is not theory you bolt on later; it is the way C# is designed.

The four pillars of OOP

We cover all four across L10 and L11: Encapsulation (hiding internals, today), Abstraction (modelling only what matters, today), Inheritance (extending classes, L11), Polymorphism (one interface, many forms, L11).

💡 Key Point

OOP bundles data and the functions that modify it into one unit. The data can't be accidentally changed by unrelated code, and the object itself knows how to do its own work.

2. Classes vs Objects — Definition and Mental Model

A class is a template that describes what shape an object will have. Writing class Car { ... } does not create a car — it describes what a car is. Think of a class like a cookie cutter: it defines the shape, but it is not a cookie itself.

Declaring a class with properties and methods
C# — A class is just a blueprint Program.cs 
class Car
{
    public string Brand { get; set; }
    public int    Year  { get; set; }
    public int    Speed { get; set; }

    public void Accelerate(int amount) => Speed += amount;
    public void Brake(int amount)      => Speed = Math.Max(0, Speed - amount);

    public void PrintStatus()
        => Console.WriteLine($"{Brand} ({Year}) — {Speed} km/h");
}

An object is a concrete instance built from a class at runtime using new. You can create many objects from one class and each is independent — changing car1.Speed does not affect car2.Speed.

Creating independent objects from one class
C# — Two objects, same class, independent data Program.cs 
Car car1 = new Car { Brand = "Toyota", Year = 2024 };
Car car2 = new Car { Brand = "BMW",    Year = 2025 };

car1.Accelerate(60);
car2.Accelerate(100);

car1.PrintStatus();  // Toyota (2024) — 60 km/h
car2.PrintStatus();  // BMW (2025) — 100 km/h
💡 Key Point

A class is a blueprint; an object is an instance. Each new creates an independent object with its own data.

Where to place classes

In .NET 10 top-level programs, define classes after the top-level statements. For larger projects, put each class in its own file (Car.cs, Student.cs). For today's exercises, one file is fine.

3. Fields and Properties

A field is a variable inside a class, but making fields public is almost always wrong — anyone can write account.Balance = -999999 with no protection. Modern C# uses properties instead: they look like fields from the outside but use get/set accessors to control access. Think of properties like the buttons on a microwave — you don't rewire the magnetron directly; the front panel decides whether your request makes sense.

The most common form is the auto-property: you declare { get; set; } and the compiler generates the backing field for you. C# also supports read-only, init-only, and computed properties.

Auto-properties, init-only, and computed properties
C# — Property flavours in one class Program.cs 
class Product
{
    public string  Name  { get; set; }   // auto: read + write
    public decimal Price { get; set; }
    public int     Stock { get; set; }
    public string  Category { get; }  // read-only (set in ctor)
    public string  Sku  { get; init; } // init-only
    public bool    IsAvailable  => Stock > 0;       // computed
    public decimal PriceWithTax => Price * 1.12m;  // computed

    public Product(string name, decimal price, string category)
    { Name = name; Price = price; Category = category; }
}

When you need validation, write a full property with a private backing field. The set accessor receives the incoming value as value and can reject it. You can also use private set to allow the class itself to modify a property while blocking outside code.

Full property with validation and private set
C# — Controlled access with validation Program.cs 
class Person
{
    private int _age;  // backing field

    public string Name { get; set; }
    public int Age
    {
        get { return _age; }
        set
        {
            if (value < 0 || value > 150)
                throw new ArgumentOutOfRangeException("Age must be 0–150");
            _age = value;
        }
    }
    public int LoginCount { get; private set; }
    public void RecordLogin() => LoginCount++;
}
Using the Person class — what gets blocked
C# — Validation and private set in action Program.cs 
Person p = new Person { Name = "Dias" };
p.Age = 25;            // OK
p.RecordLogin();
Console.WriteLine(p.LoginCount);  // 1
// p.Age = -5;        // throws ArgumentOutOfRangeException
// p.LoginCount = 99; // ERROR — private set
💡 Key Point

Use auto-properties for simple storage, full properties for validation, init for immutable creation-time data, and => for computed values. Avoid public fields.

4. Access Modifiers

Access modifiers control who can see and modify members. The purpose is encapsulation: hide internals and expose only what callers need. Think of a vending machine — the coin slot and buttons are public; the coin-counting motor is private.

The rule of thumb: make everything private by default. Only promote a member to public when outside code genuinely needs it. This minimises the surface area for bugs.

Encapsulation in practice — public vs private
C# — Hiding internals behind a public API Program.cs 
class Wallet
{
    private decimal _cash = 0;       // hidden
    public decimal Cash => _cash; // read-only view

    public void Add(decimal amount)
    {
        if (amount <= 0) return;
        _cash += amount;
    }
}

Wallet w = new Wallet();
w.Add(100);
Console.WriteLine(w.Cash);  // 100
// w._cash = 999;  // ERROR — _cash is private
Modifier Visible to Typical use
public Everyone Properties, methods that form the public API
private Only this class Backing fields, helper methods (default if omitted)
protected This class + subclasses Data shared with derived classes (L11)
internal Same assembly/project Library classes not for external use
💡 Key Point

Default to private. Only make a member public when outside code genuinely needs access. This is the core of encapsulation.

5. Constructors — Default, Parameterised, Chaining, Primary

A constructor is a special method that runs when you create an object with new. Its job is to put the object into a valid initial state. Think of it like a phone assembly line — you never receive a phone with no screen and a note saying "please attach screen yourself."

If you write no constructor, C# provides a default one that sets everything to 0 / null / false. As soon as you write any constructor, the auto-generated one disappears. If you want both parameterless and parameterised creation, you must write both explicitly.

Parameterless and parameterised constructors
C# — Two constructors with chaining Program.cs 
class Student
{
    public string Name    { get; set; }
    public int    Age     { get; set; }
    public string Faculty { get; set; }

    public Student() { Faculty = "Undeclared"; }

    public Student(string name, int age, string faculty) : this()
    { Name = name; Age = age; Faculty = faculty; }
}
Using the Student constructors
C# — Three ways to create a Student Program.cs 
Student s1 = new Student();                               // parameterless
Student s2 = new Student("Dias", 21, "IT");               // parameterised
Student s3 = new Student { Name = "Aisha", Age = 20 };  // object initialiser

Console.WriteLine(s1.Faculty);  // Undeclared
Console.WriteLine(s2.Faculty);  // IT
Console.WriteLine(s3.Faculty);  // Undeclared

Constructor chaining (: this(...)) lets one constructor delegate to another, avoiding duplicated initialisation. For simple classes, C# 12 primary constructors put parameters directly in the class header — even less boilerplate.

Constructor chaining — Rectangle with a square shortcut
C# — One constructor calls another Program.cs 
class Rectangle
{
    public double Width  { get; }
    public double Height { get; }

    public Rectangle(double w, double h) { Width = w; Height = h; }
    public Rectangle(double side) : this(side, side) { }

    public double Area() => Width * Height;
}

Rectangle r1 = new Rectangle(4, 6);
Rectangle r2 = new Rectangle(5);  // 5×5 square
Console.WriteLine(r1.Area());  // 24
Console.WriteLine(r2.Area());  // 25
Primary constructor (C# 12) — parameters in the class header
C# — Modern shorthand Program.cs 
class Course(string title, int credits)
{
    public string Title   { get; } = title;
    public int    Credits { get; } = credits;

    public void Print() => Console.WriteLine($"{Title} ({Credits} credits)");
}

Course c = new Course("Calculus", 4);
c.Print();  // Calculus (4 credits)
💡 Key Point

Constructors guarantee objects are valid from the moment they exist. Use chaining (: this()) to avoid duplicating initialisation logic between overloads.

6. The this Keyword

Inside a class, this refers to the current instance. Think of it like the word "I" — when a Circle says this.Radius, it means "the radius of the specific circle this code is running on." Most of the time you can omit it, but you need it when a parameter name shadows a property.

Disambiguating a parameter from a property
C# — this.Radius vs radius Program.cs 
class Circle
{
    public double Radius { get; set; }

    public Circle(double radius)
    {
        this.Radius = radius;  // this.Radius = property, radius = parameter
    }

    public double Area() => Math.PI * Radius * Radius;
}

A more advanced use is the fluent interface — a method returns this so the caller can chain calls like circle.Scale(2).Area().

Fluent interface — returning this for method chaining
C# — Chaining methods with return this Program.cs 
class Circle
{
    public double Radius { get; set; }
    public Circle(double radius) { this.Radius = radius; }
    public double Area() => Math.PI * Radius * Radius;

    public Circle Scale(double factor)
    {
        Radius *= factor;
        return this;  // return the same object
    }
}

Circle c = new Circle(5);
Console.WriteLine(c.Scale(2).Area());  // 314.159... (r=10)
💡 Key Point

this refers to the current object. Use it to disambiguate parameters from properties and to chain constructors.

7. Static Members

A static member belongs to the class itself, not to any particular instance. Think of the total student enrolment at a university — it belongs to the university (the class), not to any one student (an instance). You access static members via the class name: Counter.GetTotal().

Static field and method — auto-incrementing ID
C# — Shared data via static Program.cs 
class Counter
{
    private static int _total = 0;  // shared
    public int Id { get; }

    public Counter() { _total++; Id = _total; }
    public static int GetTotal() => _total;
}
Using the Counter — each new object increments the shared total
C# — Static vs instance access Program.cs 
Counter a = new Counter();  // Id=1
Counter b = new Counter();  // Id=2
Counter cc = new Counter(); // Id=3

Console.WriteLine(Counter.GetTotal());  // 3 (class name!)
Console.WriteLine(a.Id);                 // 1 (instance)
Instance memberStatic member
Accessed via object: obj.Method()Accessed via class: Class.Method()
Each object has its own copyOne shared copy for all objects
Can use thisCannot use this
Example: car.SpeedExample: Math.Sqrt()
💡 Key Point

Static members belong to the class, not to instances. Use them for shared data (counters, configuration) and for utility methods that don't depend on instance state.

8. Value Types vs Reference Types

Classes are reference types. When you write Student s2 = s1;, you copy the reference (memory address), not the object. Both variables now point to the same data. Think of it like sharing a Google Doc link — edits by either person appear for both.

Two variables, one object — the reference trap
C# — Assignment copies the reference Program.cs 
Student s1 = new Student("Aisha", 20, "IT");
Student s2 = s1;  // same object, not a copy!

s2.Name = "Bekzod";
Console.WriteLine(s1.Name);  // Bekzod ← changed through s2!

A reference variable can also hold null — meaning "no object." Accessing a member on a null reference throws NullReferenceException. Use the ?. operator for safe access.

Null references and safe access
C# — Null check patterns Program.cs 
Student s3 = null;
// s3.Name = "Test";  // NullReferenceException!

if (s3 != null)
    Console.WriteLine(s3.Name);

// Modern null-conditional operator:
Console.WriteLine(s3?.Name ?? "(no student)");
💡 Key Point

Assigning one object variable to another copies the reference, not the object. Both variables point to the same data. This is the #1 source of unexpected behaviour for OOP beginners.

9. Putting It All Together — Complete BankAccount

This final example combines every concept from today into a single class. Think of it like a real bank teller counter — you can't reach behind the counter and change the ledger; you must ask the teller (call a method), and the teller will refuse invalid requests.

Notice the design decisions: Balance has a private set that rejects negative values. Owner and AccountNumber are read-only. The _accountCount is static because it tracks all accounts, not one account.

Step 1 — Fields, properties, and constructor
C# — BankAccount skeleton Program.cs 
class BankAccount
{
    private static int _accountCount = 0;
    private decimal _balance;

    public string Owner         { get; }
    public string AccountNumber { get; }
    public decimal Balance
    {
        get => _balance;
        private set
        {
            if (value < 0)
                throw new InvalidOperationException("Balance cannot go negative");
            _balance = value;
        }
    }

    public BankAccount(string owner, decimal initialDeposit = 0)
    {
        Owner   = owner;
        Balance = initialDeposit;
        _accountCount++;
        AccountNumber = $"ACC{_accountCount:D4}";
    }
}
Step 2 — Methods: Deposit, Withdraw, Transfer
C# — Adding behaviour to BankAccount Program.cs 
// Inside the BankAccount class:
public void Deposit(decimal amount)
{
    if (amount <= 0) throw new ArgumentException("Deposit must be positive");
    Balance += amount;
}

public void Withdraw(decimal amount)
{
    if (amount <= 0) throw new ArgumentException("Amount must be positive");
    Balance -= amount;  // setter guards against negative
}

public void Transfer(BankAccount target, decimal amount)
{
    Withdraw(amount);
    target.Deposit(amount);
}

public void PrintStatement()
    => Console.WriteLine($"[{AccountNumber}] {Owner}: {Balance:C}");

public static int GetAccountCount() => _accountCount;
Step 3 — Using BankAccount from top-level code
C# — Putting the BankAccount to work Program.cs 
BankAccount acc1 = new BankAccount("Dias", 50000m);
BankAccount acc2 = new BankAccount("Aisha");

acc1.Deposit(10000m);
acc1.Transfer(acc2, 5000m);
acc1.PrintStatement();   // [ACC0001] Dias: $55,000.00
acc2.PrintStatement();   // [ACC0002] Aisha: $5,000.00
Console.WriteLine($"Total accounts: {BankAccount.GetAccountCount()}");  // 2
💡 Key Point

Good OOP design encapsulates data and behaviour together, validates state in properties and constructors, and hides implementation details behind a clean public interface.

💻 Code Examples

Example A — Temperature Converter with Static Factory

This class demonstrates computed properties, a parameterised constructor with validation, and a static factory method that returns a new instance.

C# — Temperature class Program.cs 
Temperature boiling = new Temperature(100);
Console.WriteLine($"{boiling.Celsius}°C = {boiling.Fahrenheit}°F = {boiling.Kelvin}K");
// 100°C = 212°F = 373.15K

Temperature bodyTemp = Temperature.FromFahrenheit(98.6);
Console.WriteLine($"{bodyTemp.Celsius:F1}°C");  // 37.0°C

class Temperature
{
    public double Celsius { get; }

    // Computed properties — not stored, derived
    public double Fahrenheit => Celsius * 9.0 / 5 + 32;
    public double Kelvin     => Celsius + 273.15;

    public Temperature(double celsius)
    {
        if (celsius < -273.15)
            throw new ArgumentException("Below absolute zero");
        Celsius = celsius;
    }

    // Static factory method — alternative way to create a Temperature
    public static Temperature FromFahrenheit(double f)
        => new Temperature((f - 32) * 5 / 9);
}

Example B — Inventory Item with Private Set

This class shows how private set forces callers to use methods instead of directly modifying data. You can't just set item.Quantity = 999 — you have to go through Restock() or Sell(), which enforce business rules.

C# — Inventory item Program.cs 
InventoryItem item = new InventoryItem("Widget", 9.99m, 50);
item.Sell(3);
item.Restock(20);
item.Print();  // Widget — $9.99, qty: 67

class InventoryItem
{
    public string  Name     { get; }
    public decimal Price    { get; set; }
    public int     Quantity { get; private set; }

    public InventoryItem(string name, decimal price, int qty)
    {
        Name     = name;
        Price    = price;
        Quantity = qty;
    }

    public void Sell(int count)
    {
        if (count > Quantity)
            throw new InvalidOperationException("Not enough stock");
        Quantity -= count;
    }

    public void Restock(int count)
    {
        if (count <= 0) throw new ArgumentException("Must restock positive amount");
        Quantity += count;
    }

    public void Print()
        => Console.WriteLine($"{Name} — {Price:C}, qty: {Quantity}");
}

Task 1 Library Book Easy

Create a Book class with properties: Title (string), Author (string), Year (int), IsAvailable (bool, default true).

  • Add a constructor that takes title, author, and year.
  • Add a Borrow() method that sets IsAvailable to false (print an error if already borrowed).
  • Add a Return() method that sets it back to true.
  • Add a PrintInfo() method that prints all details.
  • In the main program, create 3 books, borrow 2, and print all 3.
💡 Hint

Use a parameterised constructor. IsAvailable should default to true in the constructor. In Borrow(), check if (!IsAvailable) first.

Task 2 Simple Counter Easy

Create a Counter class with a private _count field (default 0). Add methods: Increment(), Decrement() (cannot go below 0), Reset(), and a read-only property Value.

  • The constructor accepts an optional starting value (default 0).
  • Demonstrate: create a counter at 5, increment twice, decrement once, print the result (6).
💡 Hint

The Value property only needs a get that returns _count. In Decrement(), use Math.Max(0, _count - 1) or an if check.

Task 3 Temperature Converter Medium

Create a Temperature class that stores a value in Celsius.

  • Add computed properties Fahrenheit and Kelvin.
  • Validate that Celsius is not below –273.15 (absolute zero).
  • Add a static method FromFahrenheit(double f) that returns a new Temperature.
  • Print a conversion table for –40°C to 100°C in steps of 20.
💡 Hint

Formulas: F = C × 9/5 + 32, K = C + 273.15. The static method creates a new Temperature using the inverse formula: C = (F – 32) × 5/9.

Task 4 Grade Tracker Medium

Create a GradeTracker class that stores a student's name and an array of up to 20 grades (doubles). Implement:

  • AddGrade(double grade) — validates 0–100, adds to array.
  • Average — computed property returning the average.
  • Highest / Lowest — computed properties.
  • PrintReport() — prints name, all grades, average, highest, lowest.
  • Use a private _count field to track how many grades have been added.
💡 Hint

Declare double[] _grades = new double[20]; and int _count = 0; as private fields. In AddGrade, store at _grades[_count] and increment _count. For Average, sum the first _count elements and divide.

Task 5 Shopping Cart Hard

Design two classes: Product (Name, Price, Category) and Cart (holds up to 10 products in an array).

  • Cart.AddItem(Product) — adds a product (error if full).
  • Cart.RemoveItem(string name) — removes the first matching product.
  • Cart.GetTotal() — returns the sum of all prices.
  • Cart.PrintReceipt() — lists all items, subtotal, discount, and final total.
  • Discount rule: if subtotal > 10,000, apply 10% off.
  • You must decide: which properties should be read-only? Should Cart expose its internal array? How do you handle removal (shift elements, or use null gaps)?
💡 Hint

Keep the Product array private. Use a _count field. For removal, find the index, shift subsequent elements left by one, and decrement _count. Make Product properties read-only (set only via constructor) to prevent price tampering after adding to cart.

Task 6 Vehicle Fleet Manager Hard

Design a Vehicle class (Make, Model, Year, Mileage) and a Fleet class that manages up to 20 vehicles.

  • Vehicle: Make and Year are immutable. Mileage can only increase (via a Drive(int km) method). Add a computed property Age.
  • Fleet: Add(Vehicle), FindByMake(string) (returns all matches), GetOldest(), GetTotalMileage(), PrintAll().
  • Design decisions you must make: How do you prevent mileage rollback? Should Fleet return Vehicle references or copies? How does FindByMake return multiple results without List (use an array)?
💡 Hint

Make Mileage have private set and only modify it in Drive(). For FindByMake, first count matches, then create a correctly-sized array and fill it. Vehicle's Year should use { get; } (read-only auto-property) set in the constructor.

⚠️ Common Mistakes

Forgetting new — using a null reference
Writing Student s; declares a variable but creates no object — s is null. Calling s.Name throws a NullReferenceException. Fix: always write Student s = new Student();.
Using public fields instead of properties
public int Age; (a field) looks similar to public int Age { get; set; } (a property), but the field has no validation, no private set, and cannot be made init-only later. Always use properties.
Confusing assignment with copying
Student copy = original; does NOT create a new object — both variables point to the same instance. Changing copy.Name also changes original.Name. To truly copy, create a new object and copy each property.
Calling a static member on an instance
Writing myCounter.GetTotal() when GetTotal() is static causes a compile error. Static members are accessed via the class name: Counter.GetTotal().
Expecting a default constructor after writing a parameterised one
If you define public Student(string name) { ... }, the auto-generated parameterless constructor disappears. Writing new Student() now fails. If you need both, explicitly add public Student() { }.
Primary constructor parameters not assigned to properties
In class Person(string name), the parameter name exists but is not automatically a property. You must write public string Name { get; } = name; to expose it. Without this, outside code cannot access the value.
No validation in constructors or setters
Allowing new BankAccount("", -5000) without checking produces invalid objects that break later. Validate early: check for null/empty strings and negative values in the constructor or property setter.

🎓 Instructor Notes

⚡ Teaching tips for this lesson
  • Time allocation (80 min): Intro + OOP motivation: 8 min · Classes/Objects theory: 10 min · Live-code BankAccount from scratch: 15 min · Properties + access modifiers: 12 min · Constructors + this: 10 min · Static members + reference types: 10 min · Student practice (Tasks 1–2 in pairs): 10 min · Wrap-up + assignment intro: 5 min.
  • Start with the analogy before touching any code. Draw a cookie cutter on the board. Ask: "If I change the sprinkles on this cookie, does this other cookie change?" Wait for them to answer. This mental model must be solid before a single line of C# appears, otherwise the syntax won't stick.
  • The live-code BankAccount is the centrepiece — do it in stages. Start with public fields only. Create an account, then ask a student: "Set the balance to negative one million. What happens?" They'll do it and realise nothing prevents it. Now refactor to a property with validation, live. Show the before and after. This narrative arc is far more powerful than showing final code.
  • Deliberately make a typo when typing the constructor. Forget a semicolon or misspell a property name. Let the red squiggle appear, read the error aloud, fix it calmly. This normalises mistakes and keeps your students relaxed. Many of them are terrified of compiler errors; seeing you make one and fix it casually says "this is normal."
  • The reference-type trap will cause confusion. You must demo it live: create one Student, assign to two variables, change through one, observe the other changed. Draw a memory diagram on the board with boxes for "stack" and "heap." Ask: "How many Student objects are in memory right now?" — the answer is one.
  • Static members confuse beginners. When someone asks "when should I use static?" answer: "When the data or method does not depend on any particular object's state. A running count of all students? Static. One student's GPA? Not static."
  • Discussion questions to throw at the class when it goes quiet: "If I make an Age property and someone sets it to –50, whose fault is that — the class designer or the caller?" · "Can I create 1,000 Car objects from one Car class without writing new code?" · "Why would I ever make something private when public is more flexible?"
  • If you're running behind, cut the static constructor example and the fluent interface pattern — those are nice-to-know, not essential. Constructor chaining will eat 5+ minutes if you're not careful; cut the Rectangle example if time is tight.
  • Hands-on pause: After the main theory, stop and let your students work on Task 1 (Library Book) in pairs for 8–10 minutes. Walk around and listen — you'll hear real confusion you can address live. When you regroup, ask a pair to share their solution and discuss design choices.

📖 Lesson Summary

  • OOP bundles data (properties) and behaviour (methods) into objects, making code safer and scalable.
  • A class is a blueprint; an object is an instance created with new.
  • Use properties (auto, validated, init-only, computed) instead of public fields.
  • Access modifiers (private, public) enforce encapsulation.
  • Constructors guarantee valid initial state. Chain with : this().
  • Static members belong to the class, not instances — use for shared data.
  • Reference types: assigning one object variable to another copies the reference, not the object.
