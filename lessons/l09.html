Task 1 — General (5 pts)

What was tested: basic variable declaration, Console.ReadLine(), int.Parse(), string interpolation, Console.WriteLine().
C# — Task 1: Reference Solution Program.cs 
Console.Write("Enter your name: ");
string name = Console.ReadLine()!;

Console.Write("Enter your age: ");
int age = int.Parse(Console.ReadLine()!);

Console.Write("Enter your city: ");
string city = Console.ReadLine()!;

Console.WriteLine($"\nHello! My name is {name}.");
Console.WriteLine($"I am {age} years old and live in {city}.");
Console.WriteLine($"In 5 years I will be {age + 5}.");
Common mistake: concatenating age instead of parsing
Many students wrote string age = Console.ReadLine()!; and then "In 5 years: " + age + 5 → output: "In 5 years: 215" (string concat, not addition). Fix: always parse numeric input with int.Parse() before arithmetic.

Task 2 — IF / ELSE (7 pts)

What was tested: if/else if/else chains, comparison operators, logical operators, correct condition ordering.
C# — Task 2: Reference Solution (Grade Classifier) Program.cs 
Console.Write("Enter score (0-100): ");
int score = int.Parse(Console.ReadLine()!);

string result = score switch
{
    < 0 or > 100 => "Invalid score",
    >= 90        => "A — Excellent",
    >= 75        => "B — Good",
    >= 60        => "C — Passed",
    _             => "F — Failed"
};

Console.WriteLine($"Score {score}: {result}");
Wrong condition order — broad range catches everything first
if (score >= 60) placed before if (score >= 90) means a score of 95 always prints "C — Passed". The specific conditions must come first. This was the most common mistake across all submissions.
Missing the invalid input check
Entering -5 or 200 caused the program to print a grade as if it were valid. Always validate input at the boundary — the very first check should be the guard clause.

Task 3 — Loops (2 pts)

What was tested: for loop, while loop, break/continue, off-by-one awareness.
C# — Task 3: Reference Solution (FizzBuzz) Program.cs 
for (int i = 1; i <= 50; i++)
{
    // Check combined condition FIRST
    if      (i % 15 == 0) Console.WriteLine("FizzBuzz");
    else if (i % 3  == 0) Console.WriteLine("Fizz");
    else if (i % 5  == 0) Console.WriteLine("Buzz");
    else                   Console.WriteLine(i);
}
Checking % 3 and % 5 separately with two ifs — prints both Fizz AND Buzz
Using two separate if blocks (not if/else if) means 15 prints both "Fizz" and "Buzz" on separate lines instead of "FizzBuzz". Use else if chaining, and check the combined condition (% 15 == 0) first.

Task 4 — Loops, Arrays & Methods (6 pts)

What was tested: Arrays, for/foreach, user-defined methods with array parameters, return values, basic algorithm logic.
C# — Task 4: Reference Solution (Array Stats + Filtering) Program.cs 
int[] data = ReadArray(6);

Console.WriteLine($"Sum     : {Sum(data)}");
Console.WriteLine($"Average : {Average(data):F2}");
Console.WriteLine($"Max     : {Max(data)}");
Console.Write("Evens   : ");
PrintEvens(data);

static int[] ReadArray(int n)
{
    int[] a = new int[n];
    for (int i = 0; i < n; i++)
    {
        Console.Write($"[{i}] ");
        a[i] = int.Parse(Console.ReadLine()!);
    }
    return a;
}

static int    Sum(int[] a)     { int s=0; foreach(int n in a) s+=n; return s; }
static double Average(int[] a) => (double)Sum(a) / a.Length;
static int    Max(int[] a)
{
    int m = a[0];
    foreach (int n in a) if (n > m) m = n;
    return m;
}
static void   PrintEvens(int[] a)
{
    foreach (int n in a) if (n % 2 == 0) Console.Write(n + " ");
    Console.WriteLine();
}
Everything in Main with no methods
Several submissions were a single 60+ line block with no methods at all. The task explicitly required methods — this received partial credit. In OOP, methods become mandatory, so it is essential to practice them now.
Calling Array.Sort() on the input before computing stats
Some students sorted the array at the start to find min/max easily — which is valid — but then the original order was lost. Min and max should be found by a single pass with comparisons, not by sorting (sorting is O(n log n); a single pass is O(n)).

💬 Q&A — Frequently Asked Questions

When should I use var vs. an explicit type?

Use var when the type is obvious from the right side and repeating it adds no information:

var name = "Alice"; — clearly a string, var is fine.

var result = Calculate(); — what type does Calculate() return? Unknown without checking. Use the explicit type here.

Rule of thumb: if a reader can't tell the type at a glance, write it out explicitly.

What is the difference between == for strings and for numbers?

For int, double, and other value types, == compares the values directly. Simple.

For string, == in C# also compares the content of the string (not memory address — unlike Java). So "hello" == "hello" is true.

This is a C#-specific behaviour. In Java, you need .equals() for string comparison. In C#, == works correctly for strings.

My program crashes with FormatException — what does that mean?

It means you called int.Parse() on a string that is not a valid integer — for example, the user typed letters, a decimal number, or left the input blank.

Fix: use int.TryParse() and check the return value, or add a try/catch block (covered later in the course).

C#
if (int.TryParse(Console.ReadLine(), out int n))
    Console.WriteLine($"You entered: {n}");
else
    Console.WriteLine("Not a valid number.");
What changes in Part 2 — OOP? Is it harder?

Part 2 introduces classes and objects — a way of grouping related data and methods together. Instead of scattered variables and functions, you model real-world concepts as objects.

It is conceptually different, not necessarily harder. The syntax you already know (variables, methods, loops, arrays) all still applies — you just organise it differently.

The key new ideas coming in L10–L11:

  • Class — a blueprint (e.g., Student)
  • Object — an instance of a class (e.g., new Student())
  • Fields and Properties — data that belongs to the object
  • Constructor — a special method that initialises the object
  • Inheritance — one class extending another
